# Basis Protocol MCP Server — Complete Spec

**Version:** 1.0.0  
**Date:** March 23, 2026  
**Repo:** `basis-mcp` (separate from Deploy-Guide)  
**Narrative:** *Before any agent executes a financial transaction, it can verify the risk state of every asset and counterparty through Basis.*

---

## 1. Architecture

### Separation Principle

The MCP server is a **separate project** (`basis-mcp`) that consumes the Basis API over HTTPS. It has:

- No database access
- No internal imports from the Basis codebase
- No shared runtime with the scoring engine
- Its own npm package, its own repo, its own deployment

This is deliberate. The MCP server is a **distribution channel**, not an extension of the scoring engine. It proves the thesis that any system can consume Basis as infrastructure — the MCP server itself is the first "external" consumer.

### System Diagram

```
┌──────────────────────────────────────────────────────┐
│  AI Agent (Claude, GPT, Cursor, ElizaOS, etc.)       │
│                                                      │
│  "Should I execute this swap?"                       │
│  "What's the risk profile of this counterparty?"     │
│  "Which stablecoins in my portfolio are at risk?"    │
└──────────────────┬───────────────────────────────────┘
                   │ MCP Protocol (JSON-RPC over stdio/HTTP)
                   ▼
┌──────────────────────────────────────────────────────┐
│  basis-mcp (this server)                             │
│                                                      │
│  8 tools exposed via MCP SDK                         │
│  Transforms Basis API responses into agent-friendly  │
│  structured data with risk interpretations           │
└──────────────────┬───────────────────────────────────┘
                   │ HTTPS (fetch)
                   ▼
┌──────────────────────────────────────────────────────┐
│  Basis Protocol API                                  │
│  https://basis-demo.replit.app/api/*                 │
│                                                      │
│  /api/scores         — all SII scores                │
│  /api/scores/{coin}  — detail breakdown              │
│  /api/methodology    — formula + weights             │
│  /api/wallets/{addr} — wallet risk profile           │
│  /api/wallets/riskiest — most at-risk wallets        │
│  /api/backlog        — unscored asset queue          │
└──────────────────────────────────────────────────────┘
```

### Transport

The server supports both transports:

- **stdio** — for local integrations (Claude Desktop, Cursor, Claude Code)
- **Streamable HTTP** — for remote/hosted integrations (cloud agents, registry listings)

The hosted HTTP endpoint enables any agent framework (AgentKit, ElizaOS, Olas) to connect without local installation.

### 2.1 `get_stablecoin_scores`

**Purpose:** Get current SII scores for all scored stablecoins.  
**When an agent uses this:** Before making any decision involving stablecoins — portfolio assessment, swap routing, collateral evaluation.

**Input Schema:**

```json
  "properties": {

    },
    "sort_by": {
      "type": "string",
      "description": "Sort order for results",
      "enum": ["score_desc", "score_asc", "name"],
      "default": "score_desc"
    }
  },
  "required": []
}
```

**Output Schema:**
```json
{
  "stablecoins": [

      "name": "USD Coin",
```

### 2.7 `check_transaction_risk`

**Purpose:** Composite risk assessment for a proposed transaction — evaluates the asset being transferred, the sender's wallet risk, and the receiver's wallet risk. This is the decision integrity tool.  
**When an agent uses this:** Before executing any stablecoin transfer, swap, or deposit. The core agent decision gate.

**Input Schema:**
```json
{
  "type": "object",
  "properties": {
    "from_address": {
      "type": "string",
      "description": "Sender's Ethereum wallet address"
    },
    "to_address": {
      "type": "string",
      "description": "Receiver's Ethereum wallet address"
    },
    "asset_symbol": {
      "type": "string",
      "description": "Symbol of the stablecoin being transferred (e.g., 'usdc')"
    }
  },
  "required": ["from_address", "to_address", "asset_symbol"]
}
```

**Output Schema:**
```json
{
  "transaction_risk": {
    "overall_assessment": "MEDIUM_RISK",
    "recommendation": "Proceed with awareness — asset is well-scored but receiver wallet has coverage gaps.",
    "risk_factors": [
      "Asset USDC has strong SII score (88.6, B+)",
      "Sender wallet is well-diversified (B+ concentration grade)",
      "Receiver wallet has 22% unscored stablecoin exposure"
    ]
  },
  "asset": {
    "symbol": "USDC",
    "sii_score": 88.6,
    "sii_grade": "B+",
    "is_scored": true,
    "weakest_category": "distribution"
  },
  "sender": {
    "address": "0xaaaa...1111",
    "risk_score": 85.2,
    "risk_grade": "B+",
    "total_stablecoin_value": 5000000.0,
    "coverage_quality": "full",
    "found_in_index": true
  },
  "receiver": {
    "address": "0xbbbb...2222",
    "risk_score": 68.4,
    "risk_grade": "C+",
    "total_stablecoin_value": 12000000.0,
    "coverage_quality": "partial",
    "unscored_pct": 22.0,
    "found_in_index": true
  },
  "coverage_gaps": [
    "Receiver holds $2.64M in unscored stablecoins (22% of portfolio)"
  ],
  "timestamp": "2026-03-23T12:00:00Z"
}
```

**Risk Assessment Logic (computed in MCP server, not in API):**
```
OVERALL_ASSESSMENT rules:
- LOW_RISK: asset scored A- or above AND both wallets scored B or above AND both wallets have 'full' or 'high' coverage
- MEDIUM_RISK: asset scored B- or above AND at least one wallet has coverage gaps OR concentration risk
- HIGH_RISK: asset scored C+ or below OR either wallet scored C or below OR either wallet not found in index
- UNKNOWN: asset not scored by SII
```

**API calls:** `GET /api/scores/{asset_symbol}` + `GET /api/wallets/{from_address}` + `GET /api/wallets/{to_address}` (three parallel calls)

## 3. Claude Code Build Prompt

Copy this prompt verbatim into Claude Code to build the server:

```
Read these files first:
- https://raw.githubusercontent.com/modelcontextprotocol/typescript-sdk/main/docs/server.md
- https://modelcontextprotocol.io/docs/develop/build-server (TypeScript tab)

You are building an MCP server for Basis Protocol — decision integrity infrastructure for on-chain finance.

PROJECT SETUP:
- Create a new project directory: basis-mcp
- Initialize: npm init -y
- Install: npm install @modelcontextprotocol/sdk zod@3 node-fetch
- Install dev: npm install -D @types/node typescript
- Set package.json type: "module"
- Add bin: { "basis-mcp": "./build/index.js" }
- tsconfig: target ES2022, module Node16, moduleResolution Node16, outDir ./build, rootDir ./src, strict true

The server consumes the Basis Protocol API over HTTPS. Base URL: https://basis-demo.replit.app
NO database access. NO internal imports. HTTPS fetch only.

FILES TO CREATE:

1. src/config.ts
   - BASE_URL constant
   - Grade ordering map for comparisons
   - Risk assessment thresholds
   - Type definitions for all API responses

2. src/api.ts
   - fetchScores(): calls GET /api/scores
   - fetchScoreDetail(coin: string): calls GET /api/scores/{coin}
   - fetchWalletProfile(address: string): calls GET /api/wallets/{address}
   - fetchRiskiestWallets(limit: number): calls GET /api/wallets/riskiest?limit={limit}
   - fetchBacklog(limit: number): calls GET /api/backlog?limit={limit}
   - fetchMethodology(): calls GET /api/methodology
   - All functions: handle errors gracefully, return typed responses, timeout after 10s

3. src/tools.ts
   - Register 8 tools on the McpServer using server.registerTool():

   a) get_stablecoin_scores
      - Optional inputs: min_grade (string), sort_by (string enum)
      - Calls fetchScores(), filters by grade if specified, sorts
      - Returns all scores with grades and category breakdowns

   b) get_stablecoin_detail
      - Required input: coin (string)
      - Calls fetchScoreDetail(coin)
      - Adds weakest_category and strongest_category computed fields
      - Returns full breakdown including structural subscores

   c) get_wallet_risk
      - Required input: address (string, must start with 0x)
      - Calls fetchWalletProfile(address)
      - Adds risk_interpretation as a natural language summary
      - Adds concentration.interpretation
      - Adds coverage.interpretation
      - Returns composite risk profile

   d) get_wallet_holdings
      - Required input: address (string)
      - Calls fetchWalletProfile(address), extracts holdings array
      - Adds risk_contribution field per holding:
        - If scored: "{grade} grade. {pct}% of portfolio."
        - If unscored: "UNSCORED — no SII coverage. {pct}% exposure without risk assessment."
      - Returns detailed holdings with per-asset scores

   e) get_riskiest_wallets
      - Optional input: limit (number, default 20, max 100)
      - Calls fetchRiskiestWallets(limit)
      - Adds capital_at_risk_interpretation per wallet
      - Computes total_at_risk_capital across all returned wallets

   f) get_scoring_backlog
      - Optional input: limit (number, default 20, max 100)
      - Calls fetchBacklog(limit)
      - Adds coverage_gap_interpretation per asset
      - Computes total_unscored_value

   g) check_transaction_risk
      - Required inputs: from_address, to_address, asset_symbol
      - Makes 3 parallel API calls: fetchScoreDetail(asset), fetchWalletProfile(from), fetchWalletProfile(to)
      - Computes overall_assessment: LOW_RISK, MEDIUM_RISK, HIGH_RISK, or UNKNOWN
      - Assessment rules:
        LOW_RISK = asset A- or above AND both wallets B or above AND coverage full/high
        MEDIUM_RISK = asset B- or above AND some coverage gaps
        HIGH_RISK = asset C+ or below OR wallet C or below OR wallet not found
        UNKNOWN = asset not scored
      - Generates human-readable recommendation and risk_factors array
      - Returns composite assessment with all three risk dimensions

   h) get_methodology
      - No inputs
      - Calls fetchMethodology()
      - Adds verification object confirming: public methodology, deterministic scores, no customer-specific adjustments
      - Returns full methodology with wallet scoring details

4. src/index.ts
   - Creates McpServer instance: name "basis-protocol", version "1.0.0"
   - Capabilities: tools only (no resources or prompts for V1)
   - Server description: "Decision integrity infrastructure for on-chain finance. Verifiable risk scores for stablecoins and wallet risk profiles."
   - Imports and calls registerTools(server) from tools.ts
   - Creates StdioServerTransport and connects
   - Console.error for logging (never console.log — breaks stdio)

5. src/prompts.ts (optional, V1.1)
   - Pre-built prompt templates for common agent workflows:
     - "pre_transaction_check" — runs check_transaction_risk with context
     - "portfolio_risk_report" — runs get_wallet_risk + get_wallet_holdings
     - "market_risk_overview" — runs get_stablecoin_scores + get_riskiest_wallets

TOOL ANNOTATIONS (add to every tool registration):
- readOnlyHint: true (all tools are read-only)
- openWorldHint: true (results depend on external API state)

ERROR HANDLING:
- If the Basis API is down, return a structured error: { error: true, message: "Basis API unavailable", retry_after_seconds: 30 }
- If a wallet is not found (404), return: { error: false, found_in_index: false, message: "Wallet not yet indexed by Basis. Only wallets with stablecoin holdings are indexed." }
- If a stablecoin is not found, return: { error: false, is_scored: false, message: "This stablecoin is not yet scored by SII. It may appear in the scoring backlog." }
- Never throw unhandled exceptions — always return structured responses

PACKAGE.JSON additions:
- name: "@basis-protocol/mcp-server"
- description: "MCP server for Basis Protocol — verifiable risk intelligence for stablecoins and wallet risk profiles"
- keywords: ["mcp", "risk", "stablecoin", "defi", "wallet", "basis-protocol"]
- repository: "https://github.com/shlok-lgtm/basis-mcp"
- license: "MIT"
- files: ["build"]

After building, test with:
npx @modelcontextprotocol/inspector build/index.js
```

## 4. MCP Registry Publishing Process

### 4.1 Official MCP Registry (registry.modelcontextprotocol.io)

This is the canonical registry. Publishing here means Basis appears in every MCP client that queries the registry.

**Steps:**

1. **Publish to npm first:**
```bash
cd basis-mcp
npm run build
npm publish --access public
```

2. **Install the publisher CLI:**
```bash
# Download from https://github.com/modelcontextprotocol/registry/releases
curl -L https://github.com/modelcontextprotocol/registry/releases/download/v1.0.0/mcp-publisher_1.0.0_linux_amd64.tar.gz | tar xz
```

3. **Create server.json in `.mcp/` directory:**
```json
{
  "$schema": "https://static.modelcontextprotocol.io/schemas/2025-07-09/server.schema.json",
  "name": "io.github.shlok-lgtm/basis-mcp",
  "version": "1.0.0",
  "description": "Decision integrity infrastructure for on-chain finance. Verifiable risk scores for stablecoins and wallet risk profiles via Basis Protocol.",
  "title": "Basis Protocol — Risk Intelligence",
  "websiteUrl": "https://basisprotocol.xyz",
  "packages": [
    {
      "registryType": "npm",
      "identifier": "@basis-protocol/mcp-server",
      "version": "1.0.0",
      "transport": {
        "type": "stdio"
      },
      "packageArguments": [],
      "environmentVariables": []
    }
  ],
  "repository": {
    "url": "https://github.com/shlok-lgtm/basis-mcp",
    "source": "github"
  },
  "remotes": [
    {
      "transportType": "streamable-http",
      "url": "https://mcp.basisprotocol.xyz/mcp"
    }
  ]
}
```

4. **Add the mcp-name marker to npm package README.md:**
```html
<!-- mcp-name: io.github.shlok-lgtm/basis-mcp -->
```

5. **Publish to registry:**

### 5.5 Methodology Verification

**User to Agent:** "How does Basis calculate stablecoin scores? Is the methodology public?"

**Agent calls:** `get_methodology`

**Agent response:** "Basis uses the Stablecoin Integrity Index (SII), version 1.0.0. The formula is:

**SII = 0.30×Peg + 0.25×Liquidity + 0.15×Flows + 0.10×Distribution + 0.20×Structural**

Peg stability is weighted highest (30%) because maintaining dollar parity is the fundamental promise. Structural risk (20%) captures reserves, smart contract security, oracle integrity, governance, and network risk.

The methodology is fully public, scores are deterministic (same inputs = same score), and there are no customer-specific adjustments. This is verified infrastructure, not advisory — Basis doesn't modify scores for clients."

### 5.6 Agent-to-Agent Risk Verification

**ElizaOS agent making autonomous trades:** Before executing a rebalance, the agent calls `check_transaction_risk` for each planned trade. If any trade returns HIGH_RISK, the agent pauses execution and requests human review. If all trades return LOW_RISK, the agent executes autonomously.

This is the decision integrity thesis — the agent doesn't need to trust its own risk assessment. It calls Basis to verify.

3. **The decision integrity gap is concrete.** `check_transaction_risk` is the proof — before any agent executes a financial decision, it can verify asset quality AND counterparty risk in one call.
4. **State accumulation starts now.** Every wallet indexed, every score computed, every backlog entry tracked — this is unreplicable historical state.
